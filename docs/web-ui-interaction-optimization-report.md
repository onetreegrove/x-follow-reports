# Web UI 与交互优化报告

## 1. 背景

本次 review 范围为 `web/` 目录下的前端报告阅读界面，主要涉及：

- `web/src/App.vue`
- `web/src/components/ReportSidebar.vue`
- `web/src/components/ReportToolbar.vue`
- `web/src/components/ReportReader.vue`
- `web/src/styles.css`

当前界面采用典型的三段式结构：

- 左侧：报告树形侧栏；
- 顶部：搜索、类型筛选、刷新；
- 主区域：Markdown 报告阅读器。

整体结构清晰，功能链路完整，适合作为内部报告查看工具继续迭代。但从 UI 细节、交互反馈、移动端体验和可访问性来看，还有较多可以提升的空间。

## 2. 当前体验概览

### 2.1 已具备能力

当前 Web 界面已经具备以下核心能力：

- 加载报告列表；
- 自动选中首个报告；
- 按月份、日期、报告类型组织侧栏；
- 支持搜索标题、摘要、账号、路径等内容；
- 支持按“早报 / 午报 / 晚报”筛选；
- 支持查看 Markdown 渲染后的报告正文；
- 支持移动端侧栏收起；
- 支持刷新报告列表。

### 2.2 当前主要问题

当前问题主要集中在以下几类：

1. **筛选与选中状态不同步**：用户搜索或筛选后，右侧仍可能显示不在当前筛选结果中的旧报告。
2. **侧栏报告项信息不足**：报告列表中每项只显示时间，难以判断报告内容差异。
3. **移动端侧栏交互不完整**：抽屉打开后缺少遮罩、点击外部关闭、Esc 关闭等常见交互。
4. **加载和错误状态反馈偏弱**：列表加载、详情加载、刷新失败等状态没有清晰区分。
5. **阅读体验还有提升空间**：Markdown 样式覆盖不完整，路径信息展示优先级偏高，文章排版还可以更聚焦。
6. **可访问性细节不足**：筛选按钮缺少 `aria-pressed`，当前选中项缺少 `aria-current`，键盘焦点态不明显。

## 3. 重点优化建议

### 3.1 筛选结果与当前报告联动

#### 问题

当前 `filteredReports` 会随着搜索词和类型筛选变化，但 `selectedId` 不会自动校正。

这会导致：

- 左侧列表中已经没有当前选中的报告；
- 右侧仍显示旧报告；
- 用户容易误以为搜索或筛选没有生效。

#### 建议

当筛选结果变化时：

- 如果当前 `selectedId` 仍在结果中，则保持不变；
- 如果当前 `selectedId` 不在结果中，则自动选中第一条筛选结果；
- 如果筛选结果为空，则清空当前报告，并显示无结果状态。

#### 预期效果

- 搜索和筛选行为更符合用户预期；
- 左侧列表和右侧内容始终保持一致；
- 减少“界面状态不可信”的感觉。

#### 优先级

**高**

### 3.2 提升侧栏报告项的信息密度

#### 问题

当前侧栏报告项只展示时间：

```text
09:00:12
```

如果同一天、同类型下有多份报告，用户很难判断每份报告的区别。

#### 建议

报告项中增加更多可识别信息，例如：

- 时间；
- 报告标题；
- 来源或账号；
- 资讯数量；
- 简短摘要。

建议展示形式：

```text
09:00:12
AI 行业早报 · 18 条资讯
```

或：

```text
09:00:12  18 条
OpenAI / Agent / 模型更新...
```

#### 预期效果

- 用户无需点击即可判断报告内容；
- 提升侧栏浏览效率；
- 搜索结果更容易理解。

#### 优先级

**高**

### 3.3 优化移动端侧栏抽屉交互

#### 问题

当前移动端侧栏采用 fixed 抽屉，但缺少以下交互：

- 背景遮罩；
- 点击遮罩关闭；
- 按 Esc 关闭；
- 打开抽屉时主内容仍然没有明显弱化；
- 侧栏打开后焦点管理不足。

#### 建议

移动端侧栏打开时：

- 显示半透明遮罩；
- 点击遮罩关闭侧栏；
- 支持 Esc 关闭；
- 抽屉打开时主内容视觉弱化；
- 选中报告后自动关闭侧栏，这一点当前已有，可保留。

#### 预期效果

- 移动端体验更符合常见 App / Web 抽屉模式；
- 降低误操作；
- 提升界面完成度。

#### 优先级

**高**

### 3.4 增强搜索体验

#### 问题

当前搜索框只有输入能力，没有进一步反馈：

- 不显示当前匹配数量；
- 没有清空按钮；
- 搜索无结果时缺少快捷操作；
- 用户无法快速判断搜索是否命中。

#### 建议

增加以下能力：

1. 搜索框右侧增加清空按钮；
2. 工具栏显示筛选结果数量，例如：

```text
共 128 条，当前显示 12 条
```

3. 无结果时展示：

```text
没有找到匹配报告
清空搜索 / 取消类型筛选
```

4. 如果报告数量未来变多，可对搜索输入增加 debounce。

#### 预期效果

- 用户可以快速理解当前列表状态；
- 搜索失败时有明确下一步；
- 降低恢复默认状态的成本。

#### 优先级

**中高**

### 3.5 拆分列表错误和详情错误

#### 问题

当前 `App.vue` 中只有一个 `error` 状态，同时用于：

- 报告列表加载失败；
- 报告详情加载失败。

这会导致错误展示位置和语义不够准确。

例如：

- 列表加载失败，却在阅读区显示错误；
- 详情加载失败时，侧栏仍然看似正常；
- 用户无法判断是列表问题还是当前报告问题。

#### 建议

拆分为：

- `listError`
- `detailError`

并分别展示：

- 列表错误显示在侧栏区域；
- 详情错误显示在阅读区；
- 刷新失败可以在工具栏或顶部提示。

#### 预期效果

- 错误信息更准确；
- 用户更容易判断问题来源；
- 后续接入 toast / notification 更方便。

#### 优先级

**中高**

### 3.6 优化加载反馈

#### 问题

当前详情加载时阅读区直接显示：

```text
正在加载报告...
```

如果用户从一个报告切到另一个报告，整个内容区域会被替换，体验略显跳跃。

#### 建议

可以分场景处理：

1. **首次加载**：显示完整 loading 状态。
2. **切换报告加载**：保留旧报告内容，同时在顶部显示细进度条或局部 loading。
3. **刷新列表加载**：刷新按钮显示“刷新中...”，并禁用重复点击。
4. **侧栏列表加载**：显示 skeleton 或“正在加载报告列表”。

#### 预期效果

- 页面切换更稳定；
- 用户不会因为短暂 loading 失去阅读上下文；
- 刷新状态更明确。

#### 优先级

**中**

### 3.7 改善 Markdown 阅读样式

#### 问题

当前 Markdown 样式主要覆盖：

- `h1`
- `h2`
- `h3`
- `a`

但报告正文可能包含：

- 列表；
- 引用；
- 表格；
- 代码块；
- 图片；
- 分割线。

这些元素如果使用默认样式，阅读体验会不稳定。

#### 建议

补充 Markdown 样式：

- `ul / ol / li`：优化间距；
- `blockquote`：增加左边框和浅色背景；
- `code / pre`：增加代码块样式；
- `table`：增加边框、表头背景、横向滚动；
- `img`：限制最大宽度；
- `hr`：弱化分割线；
- 外链 hover 和 visited 状态。

#### 预期效果

- 报告正文更像正式文章；
- 表格、引用、代码块可读性更好；
- 不同 Markdown 内容下展示更稳定。

#### 优先级

**中**

### 3.8 降低路径信息的视觉权重

#### 问题

当前报告详情顶部直接展示 `report.path`。对开发或排查来说有价值，但对日常阅读来说优先级不高。

#### 建议

将路径改为次级操作：

- 默认折叠；
- 放到“更多信息”里；
- 提供“复制路径”按钮；
- 或仅在 hover / debug 模式下显示。

#### 预期效果

- 报告正文焦点更集中；
- 顶部信息更简洁；
- 普通用户不会被文件路径干扰。

#### 优先级

**中**

### 3.9 默认展开逻辑优化

#### 问题

当前侧栏默认只展开今天对应的月份、日期和类型。

如果：

- 今天没有报告；
- 用户正在看历史报告；
- 用户通过筛选定位历史内容；

侧栏默认展开逻辑就不够友好。

#### 建议

展开优先级调整为：

1. 优先展开当前选中报告所在的月份 / 日期 / 类型；
2. 如果没有选中报告，再展开今天；
3. 搜索状态下，自动展开所有包含匹配结果的分组；
4. 选中报告时自动滚动到可见区域。

#### 预期效果

- 当前阅读位置更容易被定位；
- 历史报告浏览体验更好；
- 搜索结果呈现更直接。

#### 优先级

**中**

### 3.10 增强可访问性

#### 问题

当前界面已有基础按钮和输入框，但可访问性状态不完整。

主要包括：

- 筛选按钮只通过 class 表达 active；
- 当前报告项没有语义化选中状态；
- 搜索框缺少明确 `aria-label`；
- 键盘焦点样式不明显。

#### 建议

增加：

- 类型筛选按钮：`aria-pressed`；
- 当前报告按钮：`aria-current="true"`；
- 搜索框：`aria-label="搜索报告"`；
- 折叠按钮：保留现有 `aria-label`，并确保状态准确；
- 全局 `:focus-visible` 样式。

建议样式：

```css
button:focus-visible,
input:focus-visible,
summary:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: 2px;
}
```

#### 预期效果

- 键盘用户可正常操作；
- 屏幕阅读器用户能理解筛选和选中状态；
- 整体交互更规范。

#### 优先级

**中**

## 4. 建议实施顺序

### 第一阶段：修正核心交互一致性

目标：解决最容易造成误解的问题。

建议完成：

1. 筛选结果和当前报告联动；
2. 无结果状态优化；
3. 报告项展示标题 / 条数；
4. 列表错误和详情错误拆分。

收益：

- 搜索、筛选、阅读状态一致；
- 用户更容易理解当前界面；
- 改动范围可控。

### 第二阶段：优化移动端和阅读体验

目标：让界面从“能用”提升到“顺手”。

建议完成：

1. 移动端侧栏遮罩；
2. 点击外部关闭；
3. Esc 关闭；
4. Markdown 样式补齐；
5. 路径信息降级展示。

收益：

- 移动端体验明显提升；
- 阅读区更专业；
- 报告内容展示更稳定。

### 第三阶段：打磨细节和可访问性

目标：提升完整度和长期可维护性。

建议完成：

1. 搜索清空按钮；
2. 筛选结果数量；
3. `aria-pressed` / `aria-current`；
4. `focus-visible`；
5. 选中报告自动滚动到可见位置；
6. 搜索状态下自动展开匹配分组。

收益：

- 细节体验更完整；
- 对键盘和辅助技术更友好；
- 更适合长期作为内部工具使用。

## 5. 优先级清单

| 优先级 | 优化项 | 说明 |
| --- | --- | --- |
| 高 | 筛选后自动同步当前报告 | 避免右侧显示旧内容 |
| 高 | 侧栏报告项增加标题/条数 | 提升可识别性 |
| 高 | 移动端增加遮罩和外部关闭 | 完善抽屉交互 |
| 中高 | 搜索清空和结果数量 | 提升搜索反馈 |
| 中高 | 拆分列表错误和详情错误 | 错误展示更准确 |
| 中 | 加载状态优化 | 减少页面跳动 |
| 中 | Markdown 样式补齐 | 提升阅读体验 |
| 中 | 默认展开当前报告所在分组 | 改善历史报告浏览 |
| 中 | 可访问性增强 | 提升规范性 |
| 低 | 桌面折叠侧栏重设计 | 视后续使用频率决定 |

## 6. 推荐优先落地方案

如果只做一轮小版本优化，建议优先做这 5 项：

1. **筛选结果联动当前报告**
2. **报告项展示更多信息**
3. **移动端侧栏增加遮罩和点击外部关闭**
4. **搜索框增加清空按钮和结果数量**
5. **补充 focus-visible、aria-pressed、aria-current**

这 5 项改动能覆盖当前最明显的体验问题，且不会改变整体架构，适合作为第一轮 UI/UX 优化。
